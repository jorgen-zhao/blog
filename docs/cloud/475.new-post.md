## 云计算基础模型

### 三大服务模型

| 模型 | 代表服务 | 适用场景 |
|------|----------|----------|
| **IaaS** | AWS EC2, 阿里云ECS | 需要完全控制基础设施 |
| **PaaS** | Heroku, Google App Engine | 快速部署应用，忽略底层 |
| **SaaS** | Office 365, Salesforce | 直接使用现成服务 |

💡 我最喜欢用**PaaS**部署微服务，省去了配置Nginx和SSL证书的麻烦！不过最近在尝试**Serverless**架构，发现连服务器管理都省了...

### 部署模式对比

::: theorem
公有云 vs 私有云 vs 混合云
::`

- **公有云** 🌐：像租共享单车，便宜灵活但数据安全风险高
- **私有云** 🔒：相当于买私家车，安全但维护成本高
- **混合云** 🌈：折中方案，敏感数据放私有云，弹性需求放公有云

> 📡 实际项目中，我们采用混合云架构：核心交易系统部署在私有云，促销活动流量则弹性扩展到公有云。

## 云原生技术栈

### 容器化革命

🏗 容器化彻底改变了应用部署方式：

```bash
# Dockerfile示例
FROM node:18-alpine
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
EXPOSE 3000
CMD ["node", "server.js"]
```

**关键优势**：
- 环境一致性：开发、测试、生产环境完全一致
- 资源利用率：比虚拟机节省60%+资源
- 快速启动：秒级部署 vs 分钟级传统部署

### Kubernetes生态

🤯 第一次接触K8s时，我直接被`kubectl`命令的复杂性劝退...但坚持下来后发现：

```yaml
# Deployment示例
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
      - name: app
        image: my-image:latest
        ports:
        - containerPort: 8080
```

**核心组件**：
- **Pod**：最小部署单元
- **Service**：服务发现与负载均衡
- **Ingress**：流量入口管理

## 实战案例：电商系统架构

### 传统架构痛点

```
用户请求 → 负载均衡 → 应用服务器 → 数据库
          ↑               ↓
         CDN         文件存储系统
```

**问题**：
- 促销活动时服务器雪崩
- 数据库扩展困难
- 新功能上线需全量发布

### 云原生改造方案

```
用户请求 → CDN → API网关 → 微服务集群
                     ↓                ↓
                消息队列      分布式数据库
                     ↓                ↓
                 缓存集群      对象存储
```

**实施步骤**：
1. **容器化改造**：将单体应用拆分为微服务
2. **CI/CD流水线**：
   ```mermaid
   graph LR
   A[代码提交] --> B[镜像构建]
   B --> C[安全扫描]
   C --> D[部署测试环境]
   D --> E[自动化测试]
   E --> F[生产部署]
   ```
3. **可观测性建设**：
   - Prometheus监控
   - ELK日志分析
   - Jaeger链路追踪
